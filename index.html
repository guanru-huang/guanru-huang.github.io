面對客戶的工程師的未來

Posted on 2021-07-24

面對客戶的工程師有很多名字：Field Engineer, Customer Engineer, Solution Architect, Solution Engineer, Technical Sales, Presales, Delivery Engineer, Sales Engineer, Technical Consultant, Support Engineer, Partner Engineer等等。以下主要探討軟體業。

儘管名字眾多，但從產品販賣的週期來說，工作的目的可以分為三種：

  • 交易簽訂前(Pre-sales)：協助業務解說產品的技術特性，製作POC來驗證客戶需求的可實現性。
  • 交易簽訂後(Post-sales)：協助客戶導入產品，有效使用產品。
  • 維持客戶使用滿意度(Support)：解決客戶使用產品時遇到的問題。

尤其如果產品有以下特性，客戶工程師的重要性越大：

  • 產品本身無法完全滿足客戶需求，需求工程師客製化某些部分。這種情況有幾種可能性：
    • 產品本身不好，本來就難以符合客戶需求（笑）。
    • 客戶的需求太複雜或冷門。
  • 產品知識難以取得，客戶需要有人教導產品知識。這種情況有幾種可能性：
    • 產品無法試用且公開資訊少。
    • 產品非常複雜，整合情境多。

雲端對客戶工程師的影響： Support為主，服務的訂閱制

雲端軟體經常有以下幾種特性，導致客戶工程師做Pre-sales與Post-sales的工作比例降低。

  • 客戶不需要解說產品，直接試用：訂閱制的收費結構，客戶試用產品的門檻降低。
  • 客戶不需要導入，只需要支援：雲端軟體不受客戶IT環境限制，並集中管理軟體更新與維運。

但另一方面，Support扮演越來越重要的角色。有以下特點：

  • Support工作爆增：技術問題的複雜度與情感需求，加之海量的Ticekts，導致Support的工作會越來越多。
  • 服務與軟體一樣訂閱制收費：支援服務被集中化管理，當作“資源”販賣給客戶。

從客戶工程師的角度，工作的影響會是：案件的零碎化，不利於專業、人脈的累積。

客戶工程師不再以客戶Account為中心，而是以Tickets為中心執行工作。

打個比方，客戶工程師變得有點像是一個 API Server，需要你的時候call你一下，以每次的API Call計費，但價格便宜。對客戶來說，CP值變高。但客戶工程師則需要接越來越多的API Call。

但是，客戶工程師仍然有許多發展機會，而這則是另外一篇文章了。

藍海策略與真實能力

Posted on 2021-07-21 Edited on 2021-07-23

從以前到現在，我一直有個想法是：如何不跟別人競爭？也就是所謂的藍海策略。

原因可能在於：

  • 我有自知之名。直接跟別人競爭，我覺得自己會輸。
  • 跟別人競爭很累。我喜歡輕鬆的感覺。

大學學測時，我考了73級分，我備取13台大工管系英專班。（我是吊車尾哈哈。）

當時正取只有14名，蠻焦慮自己會不會上。但最後所有備取英專班的人都上了，因為前面正取的人都選擇財金、國企（笑）。

我印象非常深刻：級分越高，競爭越小，因為競爭者多，但位置更多。而更優秀的人，最終也只能選擇眾多機會中的一個。

換句話說，身旁的人越優秀時，其實競爭是越小的。因為比我更優秀的人，通常有更好的機會，所以會把機會留給我。他們不屑跟我競爭（笑）。

這個觀察有趣的點是：大部分人覺得進好大學或顯赫公司很競爭，但實際上候選人的Pool是很小的，所以競爭是很低的。

基於這樣的思考，我才選擇去日本跟德國唸書。因為，我感覺這些都不是我的優秀同儕會選擇的留學國家或市場ＸＤ。

選擇做B2B而不是B2C，也是因為不想跟大部分人競爭。

最近一直在想自己的優點跟強項是什麼（歡迎提供意見）：可能就是避開競爭的能力吧（笑）。

Notes of CS144 Introduction to Computer Networking

Posted on 2021-07-05 Edited on 2021-07-14 In Online Courses

playlist
CourseWebsite

This course is a great refresher of basic network concepts and terms, such as:

  • Four-layer network model (Application/ Transport/ Network/ Link)
  • Packets switching and routing
  • Transport layer: TCP, UDP, ICMP(ping and traceroute)
  • Network layer: IP, CIDR, IPv4 and IPv6
  • End-to-end Principle: Only end-to-end error check can protect file transfer. Network layer error check can prevent errors in transmissions, but not errors in storage.
  • Error detection: 3 schmes
    1. Checksum -
      • sums all 16-bits words in the packet and check the sum
      • less robust to prevent 2 bit errors that cancel out each other
    2. CRC (Cyclic Redundancy Code) -
      • Uses polynomical long division: Message + CRC / G = 0 passes
      • more robust and expensive
      • can check odd number of errors, 2 bits in error and single bursts of errors (<= c bits long)
    3. MAC (Message Authentication Code)
      • Uses cryptography c= MAC(M,s)
      • Cryptographically strong MAC means flipping one bit one bit of M causes the new c to be random 1 and 0 (no information)
      • no guaranteed error detection
      • used in Transport Layer Security (TLS)
  • Finite State Machines
    • State is a particular configuration of the system
    • Example: HTTP Request 3 page system
      1. Idle
      2. Page Requesting
      3. Request Pending
    • Example: TCP Connection 12 States
  • Flow Control: Stop and Wait Protocol
    • Problem: Sender sends more packets than the receiver can process
    • Solution: Don’t send more packets than the receiver can process; Receiver sends Sender feedback
    • Use 1-bit counter in data and acknowledgements to solve ACK delay issue
    • Assumptions
      1. Network does not duplicate packets
      2. Packets not delayed multiple timeouts
  • Flow Control: Sliding Window
    • Allows multiple packets in-flight
  • Retransmission Strategies: Go Back-N (window=1), Selective retransmission(window=N)
  • TCP Header
    • source port
    • destination port
    • sequence number
    • acknowledgement number
    • offset
    • reserved
    • control bits: U/A/P/R/S/F
    • window
    • checksum
    • urgent pointer
    • options padding
  • Connection Setup and Teardown
    • TCP Three-way Handshake: Syn, Syn+Ack, Ack
    • Connection Teardown: Both Side sent Fin to close connection; typically four way: Fin, Ack - Ack - Fin, Ack

終身學習的重要性

Posted on 2021-07-03 In 人生思考

專業分工對個人的風險

專業分工讓企業運行更有效率，並更容易在員工離職時，替換新的員工。

然而，從個人的角度，專業分工有風險：

  • 專業分工作能加強知識的深度，但無法讓知識面更廣。
  • 對創新不利。打造新產品與服務，需要複合的專業。
  • 專業可能被AI或自動化取代。
  • 人生只做一件事，很無聊。

解決辦法是：必須透過持續不斷地學習，將專業分工的風險降低。

終身學習的步驟

Step 0: 選擇自己想學的

  • 跟隨自己的好奇心
  • 觀察世界的趨勢，未來所需要的人才
  • 能與自己現有專業產生Synergy的知識

Step 1: 尋找學習素材

網路上能學習的免費素材非常多，真的非常幸福。

  • 各所大學提供的線上課程: Coursera, EdX
  • 大眾提供的教育平台: Udemy
  • 出版社的學習平台: O’reily (可以繳年費加入ACM，獲得免費Access)
  • 各種Youtube, Podcast, Blogs

Step 2: 持之以恆的學習

大部分人常放棄線上課程。以Youtube的許多線上課程為例 (Stanford CS224N)，第二堂課影片的觀看人數是第一堂的 50 % 以下，而第三堂課影片的觀看人數更低。

因此，學習新知識時，必須持之以恆：

  • 規劃出時間，盡可能每天都有學習。一天一小時。
  • 不要心急。知識的累積需要時間的。
  • 一個知識體系中，有不了解的部分是正常的。先掌握自己能懂的部分，再循序漸進解決不了解的部分。

Step 3: 累積與應用

利用自己學習的知識，具現化實際的產出：

  • 教導其他人。例如：將自己的發現與學習，寫部落格文章分享。
  • 製作產品。例如：學習iOS開發後，打造一個 iOS App。
  • 改變自己的習慣與生活。例如：學習到新的溝通技巧，應用在別人的對話中。

コントが始まる　人生には敗者はいなかった

コントが始まるという群衆劇を見終わった。涙流したり、大声で笑ったりして最後まで人物の優しい言動に癒されてきた。

優れた作品だと感心する。主演者の演技力が半端なくて、実際に存在している人々のように映っていた。脚本も非常に優秀で、二十代半の悩みと迷いを現実的に模写した。

二十七才の僕が第一話を見てからすぐに夢中になったのは、ドラマ人物の境遇にかなり共感できるから。

僕が芸人や音楽家ではないが、人生の交差点に立たされて、「夢」を放棄するか、もっと頑張るかという選択には、わかる気がする。特に、失敗者だと思われるか、同期と比べて後れているか、ここまで選んできた道が正しいかという疑問が一気に二十代に生まれる。

現実に目を逸らすことはできなくなって、将来の事を考えざるを得ないのは二十代だ。出世のために頑張っても、必ず成功するわけではなく、落ち込んだり疲れたりする時もある。諦めればいいのか。のんびりしたら、将来の自分は悔しくないか。

哲学者のように、心の中で人生の選択を熟考しなければならない。ときに、他人にアドバイスを求めて、自分の代わりに決めてほしいという気持ちも生じる。だが、最後には、自分で決めなければ、結果を受け入れられない。

結局、人生はコントの連続だけ。そう考えたら、人生には敗者いない。皆は、自分のコントを演じているだけ。つまらなくても、面白くても、自分のコントだ。

マクベスがほりこさんを救ったように、僕はこのドラマに救われて癒された。

How to Troubleshoot Kubernetes?

Posted on 2021-06-10 Edited on 2021-06-11 In DevOps , Kubernetes

How to troubleshoot Kubernetes?

When a pod is in the phase of “Failed” or in a “Running” phase but never ready, there are four general directions to troubleshoot this error: from the pod level, from the container level, from the cluster level, and from the host/node level.

Troubleshooting from the Pod level

  1. kubectl descibe <pod-name> : The first step of troubleshooting is to inspect the Pod by using the kubectl describe <pod-name> command to print the Pod object, which includes the PodStatus object and the PodCondition object within PodStatus object. The PodCondition object includes the fields of reason and message, which indicates the reason and details for the condition’s last transition.
  2. Syntax valiadation: Check whether there are typos or wrong hierarchies within the yaml files. Oftentimes, typos and wrong cases are reasons why pod creation is not successful. Use kubectl explain pod.spec or official website to view the API documentation for pods and compare the yaml files.
  3. kubectl logs <pod-name>: This command will give you the logs at the pod level. Inspect the logs and see where the error is.

Troubleshooting from the Container level

  1. kubectl exec -it <pod-name>: Enter the interactive mode of the container application and troubleshoot. It may be that the application runs and completes very quickly, causing the Pod to never be ready, as the container has already been destroyed after completion.
  2. Use container probes: Use a kind of probe, such as livenessProbe, readinessProbe and startupProbe with a certain handler ExecAction, TCPSocketAction and HTTPGetAction to internally check the container. The method is to include these probes inside the yaml file. If the probe results in Failure, you can troubleshoot and identify the root cause.

Troubleshooting from the Cluster level

  1. kubectl config current-context: to see the current cluster. It is possible that there are multiple clusters. Be sure to be in the cluster you want to troubleshoot.
  2. kubectl config view: to see the complete configuration of the cluster.
  3. kubectl get all -o wide: this prints more detailed information of created resource objects.

Troubleshooting form the node or host level

  1. free -m: to see if the pods have exhausted RAM and caused slowness of pods.
  2. kubectl delete: to delete unnecessary pods and release RAM. If possible, restart the nodes and assign more memory to the kubernetes cluster.

Increase monitoring capabilities for future troubleshooting

  1. Use tools such as fluentd,Prometheus and Heapster to collect logs and metrics for future analysis.

For more reference: Pod Lifecycle
